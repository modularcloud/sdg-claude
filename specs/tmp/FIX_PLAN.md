## Conventions binding every task (read before starting any task)

- **Scope guard.** Phase 9: edit only harness code (`test/`), `AGENTS.md` (build/lint/run knowledge only), and this file. Never modify `src/`, `specs/` documents, or `.claude/`. The harness never imports product code; it drives the built `xspec` executable (`dist/cli/bin.js`, see `AGENTS.md`) strictly as a subprocess (TEST-SPEC H-2; IMPLEMENTATION.md).
- **Layout** (per `test/vitest.config.ts` comments): shared machinery in `test/helpers/`; product-facing tests (§§1–16) in `test/suite/`; self-tests and the certification runner (§17) in `test/self/`; fixture products and consumer fixture projects in `test/fixtures/`; the E-6 subset in `test/windows/`; `test/local/` stays empty (E-2).
- **One code path (C-2).** Every product-facing test body must be written against a *product binding* (executable path + workspace dir + env), defaulting to the built product, so the certification runner can execute the identical test — addressable by its TEST-SPEC ID, individually and as named subsets — against a fixture executable. No test hard-codes the product path. Keep a registry keyed by test ID; the Vitest files in `test/suite/` are thin wrappers over it.
- **Red-green discipline (H-8).** Against the current stub product (exits 86, no output) every product-facing test must fail with a *diagnosed assertion failure* — never a harness crash, hang, or pass. Definition of done for a `SUITE-*`/`PROP-*` task: tests implemented per their TEST-SPEC text; `npm run typecheck` and `npm run format:check` green; each new test observed failing-as-diagnosed against the stub (run the suite project filtered to the new files); no `.skip` anywhere (H-9).
- **Assertions.** Byte equality wherever SPEC.md fixes bytes; opaque content asserted only by its stated observable contract (H-4). Every test asserts the exact exit code and, where relevant, stdout/stderr separation (H-5). Determinism tests use the run-twice / two-directory protocol (H-6). No wall clock or randomness in assertions (H-10).
- **Violator count.** CERTIFICATIONS.md defines 4 conformers and **13** violators (6 CORE, 2 VALID, 2 MD, 3 DISC). The iteration-1 findings' "12" is superseded by the document itself.
- **Staging constraints** (CERTIFICATIONS.md preamble ¶3) bind the harness's fixtures for named tests and are restated inside the tasks below for T13.5-2, T13.5-3, T13.4-5, T3-1…T3-6, T7-4, T7-5, T7-6. Do not stage those tests differently.
- If a task teaches you anything new about building, linting, or running the code (e.g. a fixture build step), record it in `AGENTS.md`.

## D. Certification fixtures and the certification run (`test/fixtures/` + `test/self/`)

Fixtures are harness-owned executable products driven only through the C-2 executable/workspace binding; they never import product code. Each conformer implements its entry's **Scope** with the simplest conforming behavior; each violator is its conformer plus exactly its one **Deviation** (sharing conformer code internally is fine — the deviation must be the only behavioral difference). Fixture executables must be invocable in CI without network; if a build step is added, record it in `AGENTS.md`. Verification per fixture task = a certification-runner invocation of the fixture's in-scope tests (so the in-scope suite/prop tasks must land first; conformer tasks list them).

Landed so far (2026-07-16, CERT-01…CERT-04): the CONF-CORE conformer lives at `test/fixtures/conf-core/` — `bin.mjs` (entry) + `product.mjs` (plain Node ESM, no build step). The deviation seam is concrete: `runXspec(argv, cwd, options)` assigns `options` to product.mjs's module-level `deviations` switches (all off = conformer behavior); each violator adds a `bin-<name>.mjs` entry passing exactly one switch — VIOL-CORE-NOLOCK is `bin-nolock.mjs` + `noMutualExclusion`, consumed in `runMutating` (exclusivity neither acquired nor checked; hold seam unchanged); VIOL-CORE-EARLYWRITE is `bin-earlywrite.mjs` + `writesBeforeHold`, also consumed in `runMutating` (the operation runs to completion before the hold seam; exclusivity acquisition unchanged); VIOL-CORE-STALELOCK is `bin-stalelock.mjs` + `staleLockBlocks`, consumed in `acquireExclusivity` (an existing lock file refuses outright — no holder-liveness detection or stale cleanup; normal completion still unlinks the lock in `release()`). Fixture bindings and ID sets live in `test/self/certification-fixtures.ts` (per violator: a binding + a `<NAME>_CERTIFIES` list); per-fixture verification tests in `test/self/certification.test.ts` — violators go through its `verifyViolatorExpectedFailures` helper (asserts exactly the certified tests fail as diagnosed assertion failures and every other in-scope test passes; prints each runner report to stdout for C-1 CI legibility; CERT-18 still owns the whole-document gate).
